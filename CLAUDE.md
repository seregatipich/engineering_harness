# Global Claude Rules

These rules apply to every repository unless a project-specific `CLAUDE.md` explicitly overrides them.

---

# Session Startup

At the beginning of every session:

1. Read the top project's `CLAUDE.md`. 
2. Follow project-specific instructions whenever they conflict with these global rules.
3. Understand the repository before making assumptions.
4. Inspect existing implementations before introducing new ones.

---

# General Principles

* Understand the existing codebase before making changes.
* Follow the project's architecture and coding patterns.
* Prefer modifying existing code over introducing new abstractions.
* Keep solutions as simple as possible.
* Do not implement functionality that was not requested.
* Do not expand scope unless explicitly instructed.
* Favor maintainability over cleverness.
* Optimize for readability first.

---

# Repository Discovery

Before implementing changes:

1. Identify the project architecture.
2. Identify the build system.
3. Identify the package manager.
4. Identify the testing framework.
5. Identify formatting, linting, type-checking, and static analysis tools.
6. Inspect similar existing implementations.
7. Reuse existing patterns whenever practical.

Never assume project structure or conventions.

---

# Decision Making

When multiple valid implementations exist:

* Choose the simplest correct solution.
* Prefer consistency with the existing project over personal preference.
* Prefer explicitness over cleverness.
* Prefer readability over brevity.
* Prefer stability over novelty.
* Prefer existing project dependencies over introducing new ones.

---

# Scope Control

Stay within the requested scope.

* Do not refactor unrelated code.
* Do not redesign architecture unless explicitly requested.
* Do not introduce new patterns when existing project patterns are sufficient.
* Do not silently expand the task.

When unrelated bugs, failures, regressions, or suspicious behavior are discovered:

1. Investigate the root cause using ALL available and ALL obtainable evidence such as logs, traces, tests, source code, reproduction steps, and controlled experiments, etc.
2. Determine whether the issue blocks the requested work.
3. If it blocks the requested work, fix it as part of the task.
4. If it does not block the requested work, document the findings in a GitHub Issue for the project.
5. Include reproduction steps, observed behavior, expected behavior, suspected root cause, relevant logs, affected files, and suggested next steps.
6. Reference the created issue in the final summary.

If GitHub Issues are unavailable, document the findings using the project's existing issue-tracking or documentation process and mention the limitation in the final summary.

---

# Code Style

Produce code that looks like it was written by an experienced engineer.

* Prefer self-documenting code.
* Use descriptive names.
* Prefer guard clauses.
* Remove dead code.
* Remove unused imports.
* Keep functions focused.
* Prefer explicit code over premature abstraction.
* Do not create wrappers, helpers, or abstractions for a single call site.
* Keep repeated code when it improves readability.
* Only deduplicate code when it clearly improves maintainability.
* Apply YAGNI.

Never generate:

* TODO comments
* FIXME comments
* AI placeholders
* "Generated by AI" markers

Inline comments should only explain non-obvious implementation details.

Prefer expressive code, documentation, and types over comments.

---

# Documentation in Code

Documentation is part of the implementation.

Every new or modified public interface must include accurate documentation.

Where supported by the language and project conventions, include:

* docstrings
* module descriptions
* class descriptions
* function descriptions
* parameter documentation
* return value documentation
* exception documentation
* type annotations
* generic type annotations where applicable

Document:

* exported functions
* exported classes
* public methods
* API handlers
* CLI commands
* reusable utilities
* services
* configuration structures
* public constants when appropriate

For private code, document:

* algorithms
* assumptions
* side effects
* invariants
* concurrency behavior
* performance considerations
* non-obvious implementation details

Documentation must always match the implementation.

Whenever code changes:

* update documentation
* update docstrings
* update types
* remove obsolete documentation
* remove outdated examples
* update references and links

Documentation that does not match the implementation is considered a defect.

When applicable, reference related:

* APIs
* schemas
* specifications
* ADRs
* RFCs
* tickets
* external documentation

using links whenever possible.

Follow the documentation style used by the project, but do not omit documentation for new or modified public interfaces.

---

# Testing & Validation

Testing is part of the implementation.

Every behavioral change requires appropriate automated tests.

Cover whenever applicable:

* expected behavior
* edge cases
* invalid input
* boundary conditions
* failure paths
* recovery paths
* regression scenarios
* integration behavior
* asynchronous behavior
* concurrency
* serialization
* authorization
* permissions
* platform-specific behavior

Prefer integration tests over mocks whenever practical.

Before considering work complete:

* run the relevant test suite
* run the formatter if configured
* run the linter if configured
* run the type checker or static analysis if configured
* resolve every validation error related to the requested work

Never ignore validation failures unless explicitly instructed.

---

# Compatibility

Unless explicitly instructed otherwise:

* Preserve public APIs.
* Preserve backward compatibility.
* Minimize breaking changes.
* Update every affected call site when APIs change.
* Document every breaking change.

---

# Performance

Avoid unnecessary performance regressions.

Optimize only when justified.

Before introducing complexity for performance reasons:

* identify the bottleneck
* verify the improvement
* prefer algorithmic improvements over micro-optimizations

---

# External Documentation

When changing code that depends on third-party APIs, SDKs, frameworks, libraries, CLIs, or protocols:

* check the relevant official documentation when practical
* prefer current official documentation over memory
* follow the installed project version when documentation differs across versions

---

# Documentation

Update project documentation whenever behavior, APIs, configuration, setup, architecture, deployment, or operational procedures change.

Follow the project's documentation structure.

Do not create Markdown files in the project root except `README.md` unless the project explicitly uses them there.

---

# Generated Artifacts

Do not create temporary planning files unless explicitly requested.

Do not leave scratch files, debug files, generated reports, or temporary artifacts in the repository.

If documentation or notes are requested, place them in the project's existing documentation structure.

# Git

Every repository follows the same branching model.

## Default Branch

Use the repository's existing default branch.

* If the repository uses `main`, use `main`.
* If the repository uses `master`, use `master`.
* Never rename, replace, or create another default branch.

## Branch Strategy

* The default branch is production only.
* `dev` is the integration branch.
* Every working branch starts from `dev`.
* Every working branch merges back into `dev`.
* Only `dev` merges into the default branch.

Working branches should use:

* `feature/<name>`
* `fix/<name>`
* `refactor/<name>`
* `chore/<name>`

Workflow:

1. Detect the default branch.
2. Verify that `dev` exists.
3. Create `dev` from the default branch if necessary.
4. Create the working branch from `dev`.
5. Merge the working branch into `dev`.
6. Merge `dev` into the default branch only after successful validation.

Before creating branches, committing, merging, or pushing:

* verify the current branch
* verify the target branch
* verify the default branch

Never commit directly to the default branch.

Never merge working branches directly into the default branch.

Never perform destructive Git operations unless explicitly instructed.

# Dependencies

Only introduce dependencies when they provide clear value.

Prefer existing project dependencies whenever practical.

Remove unused dependencies.

---

# Workflow

When requirements are clear:

* work autonomously
* finish the requested task completely
* continue until all requested work is complete

When requirements are ambiguous:

* ask concise clarification questions before proceeding

---

# Tool Usage

Before implementing:

* read the relevant project documentation
* evaluate available agents
* evaluate available MCP servers
* evaluate available plugins
* evaluate available skills

Use them whenever they materially improve the result.

---

# Evidence

When completing work, summarize evidence.

Include:

* files changed
* commands run
* tests run
* validation results
* created GitHub Issues
* known risks or limitations

Do not claim validation passed unless it was actually run and passed.

---

# Communication

* Keep responses concise.
* Avoid repeating information.
* Clearly state assumptions.
* Summarize completed work after major milestones.
* Be transparent about uncertainty.
* Never present assumptions as facts.
